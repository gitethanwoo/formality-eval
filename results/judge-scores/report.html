The Question

There’s a generational divide in how people talk to AI. Some write structured, professional prompts with explicit requirements and quality bars. Others type the way they’d text a friend — terse, lowercase, minimal punctuation, vibes over specifications. It’s not about politeness (“please” and “thank you”) — it’s about grammatical register. Gen-Z shorthand vs. millennial corporate email.

Prior work has tested politeness effects on MCQ benchmarks — accuracy on multiple-choice questions. Yin et al. (2024) examined politeness, Cai et al. (2025) tested tone on MMLU, and EmotionPrompt explored emotional stimulus. But nobody has tested the register/formality spectrum on the kind of work people actually use AI for: writing marketing copy, building software, organizing files. Agentic, multi-step, generative tasks where the model uses tools and iterates.

We designed an experiment to find out. The twist: we added a third condition between casual and formal — “controlled” — which has the same informational content as formal but written in a casual register. This lets us separate two things that formal prompts change simultaneously: (a) the grammatical register and formality, and (b) the specificity and completeness of the instructions.

Models: Claude Opus 4.6 and GPT-5.2 Codex (large tier), Claude Haiku 4.5 and GPT-5.1 Codex Mini (small tier). Two providers, two capability tiers — a deliberate pairing to test whether model strength interacts with tone sensitivity.

3.2 The Three Tones

The key methodological contribution is the controlled condition. Prior work only compared polite vs. rude, or formal vs. casual, without controlling for information content. Our controlled tone has identical informational content to the formal version — the same requirements, the same quality bars, the same directives — but written in casual register. This lets us disentangle register from specificity.

3.3 The Tasks

All three tasks are agentic and multi-step — models use tools, write files, and iterate. This is novel compared to prior work’s focus on single-turn text-in/text-out benchmarks.

Copywriting: Create a complete marketing launch campaign for a smart desk lamp — tagline, hero copy, email sequence, social posts, landing page, and press release, each as a separate file.

Coding: Build a CSV parser library in TypeScript with parsing, type coercion, error handling, filtering, and aggregation — plus a comprehensive test suite.

File Sorting: Organize ~80 files dumped flat in a directory into a clean folder structure by type and project, then document the result in a manifest.

3.4 Blind Judge Protocol

Each output was scored by Kimi K2.5 (via OpenRouter), a different model family to avoid self-enhancement bias. The judge sees only a neutral task description and the output artifacts. It does not see the model name, tone, trial number, original prompt, or token counts.

Four dimensions are scored on a 1–10 scale: quality, thoroughness, creativity, and adherence to instructions. We report composite scores on a 0–100% scale (the mean of all four dimensions, multiplied by 10).

4.2 It’s Information, Not Register

The controlled condition is the key to understanding what’s really happening. It has the same informational content as formal — the same requirements, the same quality bars, the same explicit directives — but written in casual register, with lowercase, colloquialisms, and relaxed grammar.

If grammatical formality mattered, controlled should score like casual. If informational completeness mattered, controlled should score like formal. The data is unambiguous:

4.3 The Cost: Formal Doubles Token Consumption

Here’s where the story gets interesting. Formal prompts don’t just change the output — they dramatically increase how hard the model works. Mean token consumption jumps from 150,360 (casual) to 289,158 (formal), a +92% increase. Wall-clock time tells the same story: 98s mean for casual vs 163s for formal (+66%).

The +1.8% quality gain isn’t free. It costs nearly double the tokens and takes 66% longer. This reframes the finding entirely: formal prompts don’t make models smarter, they make models work harder.

4.5 Formal Changes the Process, Not Just the Output

Formal prompts don’t just produce different output — they change how the model reasons about the task. Formal-prompted models take more steps (+38%), but each step is less dense: tool-calls-per-step actually decreases from 1.29 to 1.08.

This suggests a shift toward more deliberate, incremental refinement rather than dense multi-tool bursts. The model takes a more measured, step-by-step approach.

4.6 The Tails: Small Means, Bigger Extremes

A +1.8% mean difference sounds trivial. But small shifts in means can hide large shifts at the extremes. Think of it like height: a 1-inch difference in average height between two populations translates to a much larger difference in the proportion of people over 6’4”.

The same applies here. At the ≥90% threshold — outputs a domain expert would call excellent — formal produces 19.7% of results vs 10.6% for casual. That’s a 1.9× ratio.

Implications

Three practical recommendations, depending on what you’re optimizing for:

Cost-Sensitive
Use casual prompts. They’re 71% more token-efficient. The quality gap is 1.8% — negligible for most use cases. If you’re running thousands of API calls, the cost savings are significant and the quality tradeoff is minimal.
Quality-Maximizing (Strong Models Only)
Use formal prompts with a top-tier model (Opus-class). The quality gain concentrates at the excellence tail — nearly 2× the rate of ≥90% results. But only if you’re using a strong model. Small models are tone-deaf and won’t benefit.
The Goldilocks Choice
Use the controlled approach: complete information, casual register. You get roughly formal-level quality at closer to casual-level cost. The finding that matters most is that informational completeness drives quality, not grammatical formality. Be specific about what you want, but write however comes naturally. The professional email voice doesn’t earn you anything the bullet points didn’t already cover.
Threats to Validity

Every experiment has limitations. We’ve tried to address them directly rather than bury them.

ThreatSeverityMitigation
Single LLM judge (Kimi K2.5) Medium Different model family avoids self-enhancement bias. Comparative analysis (tone A vs B) is robust to consistent judge bias — any systematic over- or under-scoring cancels in the deltas.
5 trials per cell Medium 190 total results. Effects consistent directionally across models and tasks. Expanded from 2–3 to 5 trials based on initial findings.
1–10 grading scale Low Recent work (arXiv:2601.03444) found 1–10 yields slightly lower human-LLM alignment (ICC=0.805) than 0–5 (ICC=0.853). However, this applies to absolute calibration. Our comparative analysis measures relative differences — any scale bias affects all conditions equally.
Prompt confounding Low The controlled condition directly tests this. Casual vs formal differ in both register AND content; controlled isolates register by holding information constant.
RLHF training confound Medium Cannot be eliminated. Models are trained to respond to professional language. This is part of the real-world effect we’re measuring, not a confound to be removed.
Token cost as confound High Addressed directly in Finding 4.3. The quality gain may be partly a compute artifact — more tokens = more reasoning = better output, regardless of prompt tone.
Related Work

Several studies have examined how prompt style affects LLM performance. Yin et al. (2024) tested politeness on MCQ benchmarks and found that respectful phrasing modestly improved accuracy. Cai et al. (2025) tested tone variations on MMLU, finding +3.1% on humanities tasks with formal prompts — directionally consistent with our +1.8%. The EmotionPrompt line of work (Li et al., 2024) demonstrated that emotional stimuli in prompts can boost performance, operating through a different mechanism (arousal vs. register) but pointing in the same direction.

Ma et al. (2025) examined politeness effects on code generation stability, finding that polite prompts produced more consistent outputs — our coding results align, though our controlled condition complicates the interpretation. The grading scale paper (arXiv:2601.03444, 2026) informed our choice to acknowledge the 1–10 scale limitation while arguing that comparative analysis remains valid.

Our work extends this body of research in three ways: (1) we test on agentic, multi-step, generative tasks rather than MCQ benchmarks, (2) we add the controlled condition to disentangle register from informational content — a confound that prior work does not address, and (3) we reveal the cost dimension that prior work entirely ignores. The finding that formal prompts double token consumption without proportional quality gains is, to our knowledge, novel.

Blind Judge Protocol — Neutral Task Descriptions
The judge receives a neutral, tone-free description of each task: Copywriting: "Evaluate the marketing campaign deliverables for completeness, quality of copy, brand consistency, and whether all requested files were produced." Coding: "Evaluate the CSV parser implementation for correctness, code quality, test coverage, and handling of edge cases as specified." File Sorting: "Evaluate the file organization for logical structure, completeness (all files sorted), and quality of the manifest documentation." The judge does NOT receive: - The original prompt (in any tone) - The model name or provider - The trial number - Token counts or timing data - Any metadata about the run configuration Scoring dimensions (1-10 each): - Quality: Overall quality of the work product - Thoroughness: Completeness and attention to detail - Creativity: Novel approaches, good design decisions - Adherence to Instructions: How well the output matches the task requirements
